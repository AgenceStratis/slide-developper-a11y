ARIA

ARIA est une spécification technique du W3C qui a pour but d'améliorer l'accessibilité des contenus développés avec HTML, JavaScript, et les technologies connexes.

ARIA permet d'ajouter de la sémantique et des informations structurelles et comportementales, par l'utilisation de rôles, de propriétés et d'états aux éléments HTML.

  • Fournit aux aides techniques la sémantique des balises HTML, et notamment HTML5, sans utiliser les balises HTML natives.
  • Peut étendre la sémantique au-delà de ce que propose HTML.
  • Peut changer le rôle, l'état et les propriétés des éléments HTML restitués par les aides techniques.
  • Permet le support de HTML5 aux navigateurs et aides techniques anciens.

ARIA

Quand utiliser ARIA ?

  1. Pour reproduire la sémantique des éléments HTML (<nav> => role="navigation", <h1> => role="heading" aria-level="1") lorsqu'il n'est pas possible de le faire en HTML (conserver la rétrocompatibilité, satisfaire les SEO, accès limité au code...) ;
  2. Pour ajouter de la sémantique qui n'existe pas en HTML (search, tablist...) ;
  3. Pour signaler la mise à jour dynamique d'un contenu (aria-live, role="log|status|alert") ;
  4. Pour nommer des composants (carrousel, fenêtre modale) ou renommer des éléments interactifs (lien, bouton, champs) ;
  5. Pour assurer la compatibilité avec les technologies d'assistance anciennes qui ne prennent pas en charge HTML5.

ARIA

Quand utiliser ARIA ?

ARIA ne doit être utilisé que lorsque c'est nécessaire !

Toujours préférer les fonctionnalités HTML natives pour fournir la sémantique.

HTML peut parfois montrer certaines limites : contrôle limité sur le code, composant trop complexe, équivalents HTML trop complexes à implémenter... ARIA devient alors un outil précieux pour améliorer l'accessibilité.

ARIA

Les attributs aria-*

Quelques attributs de zone Live

  • Indique que l'élément sera modifié afin que les technologies d'assistance puissent "l'écouter" : aria-live
    • Avec la valeur polite, les technologies d'assistance informeront l'utilisateur sans interrompre la tâche en cours (message à priorité peu élevée).
    • Avec la valeur assertive, les technologies d'assistance avertiront immédiatement l'utilisateur pouvant effacer la file d'attente des informations non-restituées (message à priorité très élevée).
  • Indique si les technologies d'assistance présenteront la totalité (true) ou seulement certaines parties de l'élément modifié ou seulement l'enfant modifié (false) : aria-atomic
  • Indique qu'un élément est en cours de modification (true) et que les technologies d'assistance doivent attendre que les modifications soient terminées avant de les exposer à l'utilisateur (false) : aria-busy
  • ...

Enjeu pour l'utilisateur

  • Une problématique importante pour les personnes aveugles et les grands malvoyants.
  • Apporter une alternative aux images porteuses d'informations.
  • Ne pas renseigner une alternative pour les images décoratives.
  • Ne pas créer de la redondance d'information.

Les images

Prendre en compte le contexte

Si le contexte est suffisamment pertinent et explicite, l'image devra être ignorée par les technologies d'assistance.

Si une ou plusieurs informations présentes sur l'image sont absentes du contenu adjacent, alors l'image doit être décrite, ou le contenu adjacent doit être complété.

Astuce :
Cachez l'image et cherchez dans le contexte les informations véhiculées par l'image. Si ces informations sont présentes dans le contexte, pas besoin d'alternative à l'image.
Si les informations sont partielles ou absentes, alors il faudra ajouter une alternative à l'image.

Les images informatives : la balise <img>

<img src="url.png" alt="alternative de mon image">
  • L'attribut alt doit être rempli.

Cas particulier de l'attribut title

<img src="url.png" alt="" title="alternative de mon image">

Selon la configuration de certaines aides techniques, le title peut être ignoré ou lu à la place du alt. Mais son utilisation reste conforme.

  • L'attribut title peut être utilisé, à la place ou en complément.
  • Si les attributs alt et title sont remplis, les deux informations peuvent être restituées par les aides techniques.

Les images avec légende

Critère 1.9. Chaque légende d'image est-elle, si nécessaire, correctement reliée à l'image correspondante ?

Une légende est associée visuellement à son image. Les aveugles n'ayant pas cette information pourraient ne pas comprendre cette liaison.

  • Alternative : informations véhiculées par l'image.
  • Légende : informations relatives à l'image qui ne sont pas véhiculées par l'image : auteur, droit d'auteur, date, contexte de l'image

Attention : quand une légende est présente, l'alternative est obligatoire (un simple "légende ci-dessous" peut suffire).

Les images avec légende : le balisage

<figure role="figure" aria-label="Copyright 2016 auteur">
    <img alt="alternative de l'image" src="image.png"/>
    <figcaption>Copyright 2016 auteur</figcaption>
</figure>

<div role="figure" aria-labelledby="caption">
  <img src="image.png" alt="alternative de l'image">
  <p id="caption">Copyright 2016 auteur </p>

L'image et sa légende doivent être liées dans le code :

  • Utiliser les balises <figure> et <figcaption> (HTML5).
  • Utiliser les attributs role="figure" (ou role="group") sur le groupe, et nommer ce groupe avec l'attribut aria-label[ledby] :
    • pour s'assurer de la compatibilité avec les aides techniques anciennes ou non-compatibles avec HTML5
    • si l'utilisation de <figure> et <figcaption> est impossible.

Les cadres peuvent intégrer une grande variété de contenus pouvant être plus ou moins riches. Les utilisateurs de lecteur d'écran ont besoin d'être informés du type de contenu qu'ils vont rencontrer afin de savoir s'ils vont l'explorer.

Les cadres (iframe)

Des contenus tiers accessibles

Si les informations du cadre sont rédigées, financées, développées, ou sous le contrôle de l'organisme, ils doivent être accessibles.

Exemples :

  • Un formulaire de contact développé par les équipes internes.
  • Annuaire des agents développé par les équipes internes.

Si les informations du cadre ne sont ni rédigées, ni financées, ni développées, ni sous le contrôle de l'organisme, ils peuvent être exemptés du contrôle de l'accessibilité.

Cependant, vos contributions doivent respecter l'accessibilité au mieux.
Ou bien, une alternative accessible doit être fournie.

Exemples :

  • Un flux Twitter remontant les tweets rattachés à un hashtag spécifique
    • Les tweets rédigés par l'organisme doivent respecter les règles d'accessibilité (alternative des images informatives, utilisation raisonnée des émojis, liens explicites, contrastes...)
  • Une visionneuse Calaméo sur laquelle vous ne pouvez pas contrôler le code.
    • Vous devrez fournir un version alternative accessible au format web, PDF ou Word.

Les cadres (iframe)

Les cadres techniques

Certains cadres qui ne comportent pas de contenus ne sont pas à restituer aux technologies d'assistance, notamment les cadres techniques comme Google reCAPTCHA.
Dans ce cas il est nécessaire de les cacher aux technologies d'assistance.

<iframe src="…" aria-hidden="true" tabindex="-1" title="contenu technique"></iframe>
  • aria-hidden="true" : cache le contenu aux technologies d'assistance.
  • tabindex="-1" : empêche la prise de focus sur l'élément.
  • title : précise le type de contenu rencontré s'il n'est pas possible de cacher l'élément, ou s'il est tout de même lu.

Les couleurs

Certains utilisateurs ne peuvent pas percevoir les couleurs (aveugles et grands malvoyants), d'autres ont une vision partielle ou dégradée (daltoniens, vue en niveau de gris ou en noir et blanc).
Ne pas percevoir les couleurs peut être une réelle problématique dans la compréhension de contenus.

Certaines images ou mises en forme peuvent véhiculer de l'information par la couleur.

Il faut donc veiller à ce que l'information ne soit pas donnée uniquement par la couleur et qu'un autre moyen permette de donner l'information.

Les cas sont variables et l'implémentation est à faire au cas par cas.

Plusieurs formats :

  • Le plus commun : SRT (SubRIP Text) ;
  • le format du W3C : WebVTT (The Web Video Text Tracks Format) ;
  • ou encore le TTML (Timed Text Markup Language).

Les contenus multimédias

Comprendre un média temporel sans le son : créer des sous-titres WebVTT

Exemple d'un fichier VTT :

WEBVTT

STYLE
::cue(.cyan){ color: cyan; }

GAUCHE
00:00:00.000 --> 00:00:00.500
<b>Flèche gauche</b>

HAUT
00:00:01.000 --> 00:00:01.500
<c.cyan>Flèche droite</c>
  1. La chaîne de caractères WEBVTT
  2. Une ligne vide
  3. La déclaration des styles (si besoin)
  4. Un texte d'entête de commentaire optionnel
  5. Un code temporel temps de départ --> temps de fin, au format "heures:minutes:secondes,millisecondes"
  6. Le texte du sous-titrage
  7. Une ligne vide

Les contenus multimédias

Comprendre un média temporel sans le son : créer des sous-titres WebVTT

Le texte de sous-titrage :

  • Sur 1 ou 2 lignes encodées en UTF-8, 35 à 40 caractères max par ligne, affichées pendant 1 seconde min. à 10 secondes max.
  • Il est possible de formater le texte à l'aide de quelques balises HTML simples et de styles :
    • l'italique <i></i> ;
    • le gras <b></b> ;
    • le souligné <u></u> ;
    • les styles, comme en CSS.

La transcription textuelle

Critère 4.1. Chaque média temporel pré-enregistré a-t-il, si nécessaire, une transcription textuelle ou une audiodescription (hors cas particuliers) ?

La transcription textuelle

La transcription textuelle doit contenir :

  • Les dialogues ;
  • Les textes affichés à l'écran ;
  • Les images, graphiques ou animations qui apportent de l'information ;
  • Le nom et le titre des locuteurs ;
  • Le nom des objets (bâtiments, produits…) ;
  • Le lieu ;
  • Tout ce qu'il se passe dans la vidéo qui apporte de l'information ;
  • Des informations sur les paramètres de la vidéo (qualité, noir et blanc…).

La transcription textuelle

Quelques conseils pour la transcription :

  • Mentionner la musique de fond si elle est importante pour comprendre l'action en cours, l'ambiance.
  • Utiliser les lettres capitales uniquement pour informer d'un cri.
    " TU AS CASSÉ MON NOUVEL ORDINATEUR "
  • Ne censurez pas ce qui est dit vocalement comme les insultes et les gros mots.
  • Ne traduisez pas une information dite dans une autre langue.
  • Si un contenu audio est incompréhensible, l'indiquer.

Critère 4.6. Pour chaque média temporel pré-enregistré ayant une audiodescription synchronisée, celle-ci est-elle pertinente ?

Critère 5.8. Chaque tableau de mise en forme ne doit pas utiliser d'éléments propres aux tableaux de données. Cette règle est-elle respectée ?

Les tableaux de mise en forme ne doivent pas utiliser d'éléments propres aux tableaux de données.

  • Pas de titre <caption>, title, ou aria-label[ledby].
  • Pas d'entête ni de liaison avec des entêtes <th>, <thead>, role="rowheader", role="columnheader", scope, headers

  • Pas de regroupement de colonnes <colgroup>.
  • Le contenu linéarisé doit rester compréhensible.
  • La balise <table> possède un attribut role="presentation".

En somme, seulement des balises <table>, <tr> et <td> sans aucun autre attribut que role="presentation".

À noter : dans les lettres d'information, ces règles doivent être respectées. Voir la rubrique associée en fin de cours.

Les liens

Enjeu pour l'utilisateur

  • Les utilisateurs présentant une déficience visuelle et qui utilisent une synthèse vocale n'ont pas une vision globale de la page, il est donc capital de faire des liens compréhensibles.
  • Les handicapés moteurs naviguant à la voix doivent pouvoir contrôler les liens grâce à leur nom accessible. Si ceux-ci ne sont pas explicites ou manquants il devient compliqué de les activer.
  • Pour les personnes ayant une déficience cognitive, les liens peu explicites (en savoir plus, cliquez-ici, etc.) n'incitent pas à l'action. L'utilisateur ne sait pas où il va être redirigé.

Les liens

Un lien n'est pas un bouton et inversement

Un lien :

  • permet de naviguer vers une autre page
  • permet de naviguer vers un autre contenu de la page (ancre)
  • Balise <a href> ou attribut role="link"

Un bouton :

  • lit une vidéo, la met en pause, baisse le volume
  • lance un slider, le met en pause, change de slide
  • ouvre/ferme un menu, une fenêtre contextuelle, une fenêtre modale
  • soumet ou vide un formulaire
  • affiche/cache un contenu
  • active une option
  • Balise <button> ou attribut role="button" (+ JS)

Un lien emmène quelque part ≠ Un bouton fait quelque chose

Si un lien exécute une action au lieu de changer de page, l'utilisateur risque de ne pas comprendre ce qu'il se passe.

Les liens ne doivent donc pas être utilisées pour déclencher des actions. C'est le rôle des boutons <button type="button">, <button type="submit"> ou tout autre élément interactif utilisant l'attribut role="button" qui permettent de déclencher une action en JavaScript ou de soumettre un formulaire.

Les gabarits de conception du lien et du bouton :

  • Un lien est activé uniquement pas la touche Entrée.
  • Un bouton est activé à la fois par la touche Entrée et Espace.

De plus les agents utilisateur proposent des fonctionnalités différentes dans le cas d'un bouton ou d'un lien. Par exemple, un lien peut être ouvert dans un nouvel onglet.

Les liens

Intitulé de lien explicite

Critère 6.1. Chaque lien est-il explicite (hors cas particuliers) ?

Pour qu'un lien soit explicite, il est nécessaire de fournir un intitulé de lien qui identifie l'objectif du lien, sa fonction et sa destination : le titre d'une page de destination, d'une section ou le nom d'un document peuvent suffire.

D'autres éléments peuvent également entrer en jeu pour décrire l'objet d'un lien. Si le lien provoque un changement de contexte (ouverture dans un nouvel onglet), il est nécessaire de l'indiquer.

Les liens

Intitulé visible et nom accessible : l'intitulé visible

L'intitulé visible est uniquement le texte visible d'un lien ou d'un bouton, le texte entre <a> et </a>.

Dans un monde idéal, ce texte devrait toujours être visible et permettre aux personnes utilisant des reconnaissances vocales d'identifier et vocaliser rapidement la commande du lien. Cependant cela n'est pas toujours possible.

Dans ce cas les utilisateurs devront avoir accès à un nom accessible.

À noter : L'attribut title ne constitue pas un intitulé visible car il n'est pas visible sur des appareils tactiles ou à la prise de focus. C'est une solution conforme mais très peu robuste pour rendre un lien accessible.

Les liens

Intitulé visible et nom accessible : le nom accessible

Le nom accessible est le nom déterminé via l'API d'accessibilité de l'agent utilisateur.
C'est ce texte que les aides techniques (lecteur d'écran) restitueront.

Dans le cas d'un lien, le nom accessible est obtenu selon l'ordre suivant :

  • Passage de texte associé au lien avec l'attribut aria-labelledby.
  • Sinon, contenu de l'attribut aria-label.
  • Sinon, contenu du lien.
  • Sinon, contenu de l'attribut title.

Recommandation officielle du W3C : Accessible Name and Description Computation (anglais)

<a href="#">En savoir plus</a>

Nom accessible = intitulé visible = "En savoir plus"
 

<p id="heading">En savoir plus sur la ville</p>
<a aria-labelledby="heading" href="#">En savoir plus</a>

Nom accessible = attribut aria-labelledby = "En savoir plus sur la ville"
  

<a aria-label="En savoir plus sur la ville" href="#">En savoir plus</a>

Nom accessible = attribut aria-label = "En savoir plus sur la ville"
  

<a title="En savoir plus sur la ville" href="#">En savoir plus</a>

Nom accessible = intitulé visible = "En savoir plus"

Intitulé visible et nom accessible : particularité de l'attribut title

Selon la technologie d'assistance utilisée, l'attribut title n'est pas pris en charge de la même manière.

  • L'attribut title est une description complémentaire. Il est lu par les aides techniques en complément du nom accessible.
  • S'il est identique au nom accessible, il peut être ignoré par certaines aides techniques.
  • S'il est plus long que le nom accessible, il peut être lu à la place du nom accessible.
  • En présence de l'attribut aria-label[ledby], l'attribut title être souvent ignoré.

Dans tous les cas, il n'est pas visible sur des appareils tactiles ou à la prise de focus (utilisation au clavier uniquement). Les informations qu'il transmet ne doivent pas être essentielles.

Les liens

Intitulé visible et nom accessible : le nom accessible doit contenir au moins l'intitulé de lien visible

Les utilisateurs de logiciels de contrôle à la voix prononcent généralement une commande suivie d'une référence à un intitulé visible.

Pour activer un lien avec pour intitulé visible "Toutes les actualités", ils peuvent dire cliquez sur toutes les actualités. Le logiciel cherchera alors un élément cliquable dont le nom accessible contient "Toutes les actualités".

Si le nom accessible du lien ne contient l'intitulé visible, l'activation du lien ne fonctionnera pas. Le logiciel peut alors proposer un autre moyen plus complexe pour trouver le lien (mode grille).

Mauvais exemple

<a href="#" target="_blank" aria-label="Nouvelle fenêtre">Toutes les actualités</a>

Ici, le nom accessible est "Nouvelle fenêtre" et non "Toutes les actualités".

Bon exemple

<a href="actualites.html" target="_blank" aria-label="Toutes les actualités, nouvelle fenêtre">Toutes les actualités</a>

Ici, le nom accessible est "Toutes les actualités, nouvelle fenêtre". Il reprend bien l'intitulé visible, la reconnaissance vocale pourra faire la correspondance avec la demande de l'utilisateur cliquer sur toutes les actualités.

Les liens

Des liens composites

  • Composés de plusieurs contenus (titre, texte, image...)
  • Vérifier que la totalité du texte du lien (nom accessible) est bien explicite
  • Si le texte du lien est suffisamment explicite, les alternatives d'images peuvent être vides

Remarques

  • Il peut être difficile de comprendre la fonction et la destination d'un lien aussi long
  • Préférer un lien sur l'élément le plus explicite, comme le titre d'un article
  • Pour une raison ergonomique, il est tout à fait possible d'élargir la zone de clic en CSS

Les liens

Contexte du lien

Un lien explicite hors contexte est toujours préférable.

Un lien non explicite du type "cliquer ici" ou "en savoir plus" oblige l'utilisateur à interroger le contexte du lien. La navigation devient plus complexe et plus longue.

Si le lien n'est pas explicite hors contexte, le contexte doit permettre de le rendre explicite.

Accéder à la démo des liens non pertinents explicités grâce au contexte

Le contexte correspond au contenu adjacent :

  • Le contenu de la phrase dans laquelle le lien texte est présent.
  • Le contenu du paragraphe (balise <p>) dans lequel le lien texte est présent.
  • Le contenu de l'item de liste (balise <li>) ou le contenu d'un item de liste parent (balise <li>) dans lequel le lien texte est présent.
  • Le contenu du titre (balise <hx>) précédent le lien texte.
  • Le contenu de la ou les cellule(s) d'en-tête de tableau (balise(s) <th>) associée(s) à la cellule de donnée (balise <td>) dans laquelle le lien texte est présent.
  • Le contenu de la cellule de donnée (balise <td>) dans laquelle le lien texte est présent.

Contexte du lien : rendre un lien explicite dans le code

Quelques propositions :

  • Avec un attribut aria-describedby sur le lien pour améliorer la description en faisant référence à un autre texte
  • Avec un attribut aria-label sur le lien pour améliorer le nom accessible
  • Avec un attribut title sur le lien pour améliorer la description du lien
  • En déplaçant le lien sur un texte explicite, et en agrandissant la taille de la zone cliquable en CSS
