DOM et arbre d'accessibilité

Le chemin critique du rendu

Une page HTML est affichée sur un navigateur en plusieurs étapes que l'on appelle "chemin critique du rendu" :

  1. Construction du DOM (Document Object Model) : représentation des éléments
  2. Construction du CSSOM (CSS Object Model) : représentation des styles associés aux éléments
  3. Exécution du code Javascript
  4. Création de l'arbre de rendu : représente ce qui va être affiché sur la page au regard du DOM et du CSSOM
  5. Génération de la mise en page : application des CSS en fonction de la taille de la fenêtre active (ou viewport)
  6. Conversion du contenu visible final de la page en pixels

Au terme de ces 6 étapes, le fichier source HTML initial est affiché dans le navigateur.

DOM et arbre d'accessibilité

Le DOM

Le DOM est représentation du code HTML source sous la forme d'un arbre. Il permet de convertir le document HTML en un modèle objet manipulable (consulter, modifier) par différentes applications (extensions de navigateurs) ou langages (Javascript).

  • le premier parent est l'élément racine <html>
  • les branches sont les éléments imbriqués
  • les feuilles sont le contenu des éléments

Le DOM n'est pas une représentation exacte du code source HTML.

  • Corrigé par le navigateur si le code HTML n'est pas valide
  • Modifié en JavaScript, sans modifier le document HTML source

Le DOM n'est pas la page qui s'affiche dans le navigateur. Ce qui s'affiche dans le navigateur, c'est l'arbre de rendu (combinaison du DOM et du CSSOM) qui ne comprend que ce qui sera visible à l'écran (display:none).

L'arbre de rendu complet (DOM + CSSOM) est consultable dans la console des navigateurs (F12).

DOM et arbre d'accessibilité

L'arbre d'accessibilité

L'arbre d'accessibilité ressemble au DOM, mais représente les éléments d'une manière différente.
Il est destiné à être utilisés par les technologies d'assistance pour restituer le contenu d'une page.

Un objet de l'arbre d'accessibilité contient jusqu'à 4 informations pour chaque élément :

  • son nom permettant d'y faire référence
  • sa description permettant de le décrire et de compléter son nom
  • un rôle pour expliquer son utilité
  • un état, exemple coché/décoché pour une case à cocher

L'arbre d'accessibilité est consultable dans la console des navigateurs (F12). Celui de Firefox est particulièrement optimisé.

ARIA

ARIA est une spécification technique du W3C qui a pour but d'améliorer l'accessibilité des contenus développés avec HTML, JavaScript, et les technologies connexes.

ARIA permet d'ajouter de la sémantique et des informations structurelles et comportementales, par l'utilisation de rôles, de propriétés et d'états aux éléments HTML.

  • Fournit aux aides techniques la sémantique des balises HTML, et notamment HTML5, sans utiliser les balises HTML natives.
  • Peut étendre la sémantique au-delà de ce que propose HTML.
  • Peut changer le rôle, l'état et les propriétés des éléments HTML restitués par les aides techniques.
  • Permet le support de HTML5 aux navigateurs et aides techniques anciens.

ARIA

Quand utiliser ARIA ?

  1. Pour reproduire la sémantique des éléments HTML (<nav> => role="navigation", <h1> => role="heading" aria-level="1") lorsqu'il n'est pas possible de le faire en HTML (conserver la rétrocompatibilité, satisfaire les SEO, accès limité au code...) ;
  2. Pour ajouter de la sémantique qui n'existe pas en HTML (search, tablist...) ;
  3. Pour signaler la mise à jour dynamique d'un contenu (aria-live, role="log|status|alert") ;
  4. Pour nommer des composants (carrousel, fenêtre modale) ou renommer des éléments interactifs (lien, bouton, champs) ;
  5. Pour assurer la compatibilité avec les technologies d'assistance anciennes qui ne prennent pas en charge HTML5.

ARIA

Quand utiliser ARIA ?

ARIA ne doit être utilisé que lorsque c'est nécessaire !

Toujours préférer les fonctionnalités HTML natives pour fournir la sémantique.

HTML peut parfois montrer certaines limites : contrôle limité sur le code, composant trop complexe, équivalents HTML trop complexes à implémenter... ARIA devient alors un outil précieux pour améliorer l'accessibilité.

Quelques attributs de zone Live

  • Indique que l'élément sera modifié afin que les technologies d'assistance puissent "l'écouter" : aria-live

  • Indique qu'un élément est en cours de modification (true) et que les technologies d'assistance doivent attendre que les modifications soient terminées avant de les exposer à l'utilisateur (false) : aria-busy

Prendre en compte le contexte

Si le contexte est suffisamment pertinent et explicite, l'image devra être ignorée par les technologies d'assistance.

Si une ou plusieurs informations présentes sur l'image sont absentes du contenu adjacent, alors l'image doit être décrite, ou le contenu adjacent doit être complété.

Astuce :
Cachez l'image et cherchez dans le contexte les informations véhiculées par l'image. Si ces informations sont présentes dans le contexte, pas besoin d'alternative à l'image.
Si les informations sont partielles ou absentes, alors il faudra ajouter une alternative à l'image.

Des contenus tiers accessibles

Si les informations du cadre sont rédigées, financées, développées, ou sous le contrôle de l'organisme, ils doivent être accessibles.

Exemples :

  • Un formulaire de contact développé par les équipes internes.
  • Annuaire des agents développé par les équipes internes.

Si les informations du cadre ne sont ni rédigées, ni financées, ni développées, ni sous le contrôle de l'organisme, ils peuvent être exemptés du contrôle de l'accessibilité.

Cependant, vos contributions doivent respecter l'accessibilité au mieux.
Ou bien, une alternative accessible doit être fournie.

Exemples :

  • Un flux Twitter remontant les tweets rattachés à un hashtag spécifique
    • Les tweets rédigés par l'organisme doivent respecter les règles d'accessibilité (alternative des images informatives, utilisation raisonnée des émojis, liens explicites, contrastes...)
  • Une visionneuse Calaméo sur laquelle vous ne pouvez pas contrôler le code.
    • Vous devrez fournir un version alternative accessible au format web, PDF ou Word.

Les cadres (iframe)

Les cadres techniques

Certains cadres qui ne comportent pas de contenus ne sont pas à restituer aux technologies d'assistance, notamment les cadres techniques comme Google reCAPTCHA.
Dans ce cas il est nécessaire de les cacher aux technologies d'assistance.

<iframe src="…" aria-hidden="true" tabindex="-1" title="contenu technique"></iframe>
  • aria-hidden="true" : cache le contenu aux technologies d'assistance.
  • tabindex="-1" : empêche la prise de focus sur l'élément.
  • title : précise le type de contenu rencontré s'il n'est pas possible de cacher l'élément, ou s'il est tout de même lu.

Les couleurs

Certains utilisateurs ne peuvent pas percevoir les couleurs (aveugles et grands malvoyants), d'autres ont une vision partielle ou dégradée (daltoniens, vue en niveau de gris ou en noir et blanc).
Ne pas percevoir les couleurs peut être une réelle problématique dans la compréhension de contenus.

Certaines images ou mises en forme peuvent véhiculer de l'information par la couleur.

Il faut donc veiller à ce que l'information ne soit pas donnée uniquement par la couleur et qu'un autre moyen permette de donner l'information.

Les cas sont variables et l'implémentation est à faire au cas par cas.

Les couleurs

Contraste de couleur du texte et de son arrière plan

Concerne les textes, les textes en image ou les textes incrustés dans une vidéo.

Sans effet de graisse :

  • Inférieur à 24px, le ratio de contraste entre le texte et son arrière-plan doit être au moins de 4.5:1.
  • Supérieur ou égal à 24px, le ratio de contraste entre le texte et son arrière-plan doit être au moins de 3:1.

Texte en gras :

  • Inférieur à 18.5px, le ratio de contraste entre le texte et son arrière-plan doit être au moins de 4.5:1.
  • Supérieur ou égale à 18.5px, le ratio de contraste entre le texte et son arrière-plan doit être au moins de 3:1.

  7. Une ligne vide

La transcription textuelle

La transcription textuelle doit contenir :

  • Les dialogues ;
  • Les textes affichés à l'écran ;
  • Les images, graphiques ou animations qui apportent de l'information ;
  • Le nom et le titre des locuteurs ;
  • Le nom des objets (bâtiments, produits…) ;
  • Le lieu ;
  • Tout ce qu'il se passe dans la vidéo qui apporte de l'information ;
  • Des informations sur les paramètres de la vidéo (qualité, noir et blanc…).

La transcription textuelle

Quelques conseils pour la transcription :

  • Mentionner la musique de fond si elle est importante pour comprendre l'action en cours, l'ambiance.
  • Utiliser les lettres capitales uniquement pour informer d'un cri.
    " TU AS CASSÉ MON NOUVEL ORDINATEUR "
  • Ne censurez pas ce qui est dit vocalement comme les insultes et les gros mots.
  • Ne traduisez pas une information dite dans une autre langue.
  • Si un contenu audio est incompréhensible, l'indiquer.

Critère 4.6. Pour chaque média temporel pré-enregistré ayant une audiodescription synchronisée, celle-ci est-elle pertinente ?

Les liens

Enjeu pour l'utilisateur

  • Les utilisateurs présentant une déficience visuelle et qui utilisent une synthèse vocale n'ont pas une vision globale de la page, il est donc capital de faire des liens compréhensibles.
  • Les handicapés moteurs naviguant à la voix doivent pouvoir contrôler les liens grâce à leur nom accessible. Si ceux-ci ne sont pas explicites ou manquants il devient compliqué de les activer.
  • Pour les personnes ayant une déficience cognitive, les liens peu explicites (en savoir plus, cliquez-ici, etc.) n'incitent pas à l'action. L'utilisateur ne sait pas où il va être redirigé.

Les liens

Un lien n'est pas un bouton et inversement

Un lien :

  • permet de naviguer vers une autre page
  • permet de naviguer vers un autre contenu de la page (ancre)
  • Balise <a href> ou attribut role="link"

Un bouton :

  • lit une vidéo, la met en pause, baisse le volume
  • lance un slider, le met en pause, change de slide
  • ouvre/ferme un menu, une fenêtre contextuelle, une fenêtre modale
  • soumet ou vide un formulaire
  • affiche/cache un contenu
  • active une option
  • Balise <button> ou attribut role="button" (+ JS)

Un lien emmène quelque part ≠ Un bouton fait quelque chose

Si un lien exécute une action au lieu de changer de page, l'utilisateur risque de ne pas comprendre ce qu'il se passe.

Les liens ne doivent donc pas être utilisées pour déclencher des actions. C'est le rôle des boutons <button type="button">, <button type="submit"> ou tout autre élément interactif utilisant l'attribut role="button" qui permettent de déclencher une action en JavaScript ou de soumettre un formulaire.

Les gabarits de conception du lien et du bouton :

  • Un lien est activé uniquement pas la touche Entrée.
  • Un bouton est activé à la fois par la touche Entrée et Espace.

De plus les agents utilisateur proposent des fonctionnalités différentes dans le cas d'un bouton ou d'un lien. Par exemple, un lien peut être ouvert dans un nouvel onglet.

Intitulé de lien explicite

Critère 6.1. Chaque lien est-il explicite (hors cas particuliers) ?

Pour qu'un lien soit explicite, il est nécessaire de fournir un intitulé de lien qui identifie l'objectif du lien, sa fonction et sa destination : le titre d'une page de destination, d'une section ou le nom d'un document peuvent suffire.

D'autres éléments peuvent également entrer en jeu pour décrire l'objet d'un lien. Si le lien provoque un changement de contexte (ouverture dans un nouvel onglet), il est nécessaire de l'indiquer.

Intitulé visible et nom accessible : l'intitulé visible

L'intitulé visible est uniquement le texte visible d'un lien ou d'un bouton, le texte entre <a> et </a>.

Dans un monde idéal, ce texte devrait toujours être visible et permettre aux personnes utilisant des reconnaissances vocales d'identifier et vocaliser rapidement la commande du lien. Cependant cela n'est pas toujours possible.

Dans ce cas les utilisateurs devront avoir accès à un nom accessible.

À noter : L'attribut title ne constitue pas un intitulé visible car il n'est pas visible sur des appareils tactiles ou à la prise de focus. C'est une solution conforme mais très peu robuste pour rendre un lien accessible.

Le nom accessible est le nom déterminé via l'API d'accessibilité de l'agent utilisateur.

Intitulé visible et nom accessible : particularité de l'attribut title

Selon la technologie d'assistance utilisée, l'attribut title n'est pas pris en charge de la même manière.

  • L'attribut title est une description complémentaire. Il est lu par les aides techniques en complément du nom accessible.
  • S'il est identique au nom accessible, il peut être ignoré par certaines aides techniques.
  • S'il est plus long que le nom accessible, il peut être lu à la place du nom accessible.
  • En présence de l'attribut aria-label[ledby], l'attribut title être souvent ignoré.

Dans tous les cas, il n'est pas visible sur des appareils tactiles ou à la prise de focus (utilisation au clavier uniquement). Les informations qu'il transmet ne doivent pas être essentielles.

Intitulé visible et nom accessible : le nom accessible doit contenir au moins l'intitulé de lien visible

Les utilisateurs de logiciels de contrôle à la voix prononcent généralement une commande suivie d'une référence à un intitulé visible.

Pour activer un lien avec pour intitulé visible "Toutes les actualités", ils peuvent dire cliquez sur toutes les actualités. Le logiciel cherchera alors un élément cliquable dont le nom accessible contient "Toutes les actualités".

Si le nom accessible du lien ne contient l'intitulé visible, l'activation du lien ne fonctionnera pas. Le logiciel peut alors proposer un autre moyen plus complexe pour trouver le lien (mode grille).

Mauvais exemple

<a href="#" target="_blank" aria-label="Nouvelle fenêtre">Toutes les actualités</a>

Ici, le nom accessible est "Nouvelle fenêtre" et non "Toutes les actualités".

Bon exemple

<a href="actualites.html" target="_blank" aria-label="Toutes les actualités, nouvelle fenêtre">Toutes les actualités</a>

Ici, le nom accessible est "Toutes les actualités, nouvelle fenêtre". Il reprend bien l'intitulé visible, la reconnaissance vocale pourra faire la correspondance avec la demande de l'utilisateur cliquer sur toutes les actualités.

Sens de lecture

Critère 8.10. Dans chaque page web, les changements du sens de lecture sont-ils signalés ?

Le sens de lecture par défaut sur une page web est de gauche à droite.

Pour les langues s'écrivant de droite à gauche comme l'arabe ou l'hébreu, il est nécessaire de la spécifier dans le document.

Comme pour les changements de langue, les changements de sens lecture doivent être réalisés sur le document, mais aussi sur un élément de la page.

Pour indiquer un sens de lecture, il est nécessaire d'utiliser l'attribut dir sur la balise html ou body pour le document, ou sur la balise du texte dont le sens change.

Cet attribut peut prendre deux valeurs :

  • ltr (left to right) indique un sens de lecture de gauche à droite
  • rtl (right to left) indique un sens de lecture de droite à gauche. 
<span lang="ar" dir="rtl">شكرا جزيلا</span>

Zone située généralement en haut du document et contenant un logo, un slogan...

Structure du document : les zones de navigation principale et secondaire

La navigation principale est un élément que l'on retrouve sur toutes les pages du site Web. C'est celle que l'on trouve au début du contenu pour atteindre les principales pages du site.

Il existe aussi des navigations secondaires dans des sections spécifiques. Elles doivent être identifiables de la même manière que la navigation principale.

Parmi ces navigations secondaires, on retrouve :

  • le fil d'Ariane ;
  • le menu d'évitement ;
  • la pagination d'une liste de résultats ;
  • un menu de sous-rubriques ;
  • une table des matières concernant un ensemble de pages.

Les navigations sont structurées via la balise <nav>.

Structuration des éléments de liste

Critère 9.3. Dans chaque page web, chaque liste est-elle correctement structurée ?

Comme pour les titres, il est nécessaire de structurer les contenus avec les éléments appropriés. Dans le cadre d'une liste, le nombre d'éléments que contient la liste est annoncé par les technologies d'assistance, ainsi que le niveau de l'élément dans les listes imbriquées pour certaines.

Les utilisateurs utilisant un clavier peuvent ainsi éviter une liste trop longue, ou naviguer dans la liste en se rendant au début ou à la fin de celle-ci.

Structuration de l'information

Structuration des éléments de liste : Liste ordonnée et non ordonnée

Une liste est une suite d'éléments de contenu pouvant être regroupés.

  • Listes ordonnées <ol><li> : l'ordre des éléments est important pour la compréhension (ex. : recette de cuisine).
  • Listes non-ordonnées <ul><li> : l'ordre n'importe pas.

Il est également possible de structurer les listes non-ordonnées à l'aide des attributs role="list" et role="listitem".

Attention, ARIA ne permet pas de différencier la liste ordonnée de la liste non-ordonnée. Ajouter la numération manuellement.

Mise en forme exclusive avec CSS

Critère 10.1. Dans le site web, des feuilles de styles sont-elles utilisées pour contrôler la présentation de l'information ?

Selon leurs handicaps, les utilisateurs ont besoin de personnaliser l'affichage d'une page. Ils pourraient rencontrer des difficultés s'il leur est impossible de surcharger des styles.

Contenus visibles sans CSS

Critère 10.2. Dans chaque page web, le contenu visible porteur d'information reste-t-il présent lorsque les feuilles de styles sont désactivées ?

Les images de fond insérées en CSS (propriété background-image) ne doivent jamais être porteuses d'information.

Les utilisateurs ayant besoin de personnaliser l'affichage d'une page à l'aide de feuilles de styles personnalisées peuvent ne pas percevoir le contenu de ce type d'image.

Critère 10.3. Dans chaque page web, l'information reste-t-elle compréhensible lorsque les feuilles de styles sont désactivées ?

L'ordre des éléments affichés doit correspondre à l'ordre dans le DOM.

Exemples de problèmes récurrents :

Agrandissement du texte : balise meta viewport

La meta viewport ne doit pas interdire le zoom du navigateur jusqu'à 200%.

Les paramètres ci-après ne sont pas conformes :

<meta name="viewport" content="user-scalable=no">
<meta name="viewport" content="maximum-scale=1.0">
<meta name="viewport" content="maximum-scale=1.5">

Le paramètre ci-après est conforme :

<meta name="viewport" content="width=device-width, initial-scale=1, minimum-scale=1.0">

L'utilisation de user-scalable avec la valeur no empêche l'utilisateur de zoomer. Ne pas utiliser cette propriété, ou utiliser la valeur yes.

L'utilisation de maximum-scale avec une valeur inférieure à 2 empêche l'utilisateur de zoomer jusqu'à 200%. Ne pas utiliser cette propriété, ou utiliser une valeur supérieure ou égale à 2.

Visibilité du focus

Critère 10.7. Dans chaque page web, pour chaque élément recevant le focus, la prise de focus est-elle visible ?

La visibilité du focus est cruciale pour les personnes qui ont des problèmes de motricité et qui ne peuvent pas utiliser une souris ou un pointeur. Ils utilisent donc principalement un clavier qui peut parfois être spécifique à leur handicap.

  • Le focus est personnalisable par la pseudo-classe :focus et la propriété outline en CSS.
  • Le focus peut être personnalisé à condition que le ratio de contraste entre le focus et son arrière-plan soit supérieur ou égal à 3:1.

  • Utilisations interdites : outline: 0|none|transparent

À noter : l'utilisation d'une propriété autre que outline (border, background...) pour indiquer visuellement le focus n'est pas conforme.

Contenus cachés

Critère 10.8. Pour chaque page web, les contenus cachés ont-ils vocation à être ignorés par les technologies d'assistance ?

Il existe différentes techniques pour cacher des contenus spécifiques. Cependant, lorsque ces techniques sont mal utilisées, cela peut masquer du contenu aux personnes en situations de handicap et notamment aux technologies d'assistance.

Les contenus masqués via les propriétés CSS suivantes sont ignorés par les technologies d'assistance :

  • display: none;
  • visibility: hidden;
  • font-size: 0;
  • attribut aria-hidden="true"
  • attribut HTML5 hidden

Lorsque ces techniques sont employées, s'assurer que le contenu soit réellement ignoré par les aides techniques.

Scénario d'un contenu masqué qui ne doit pas être ignoré :

Une image avec lien qui, une fois survolée ou focussée, affiche du texte grâce à la propriété CSS display:none.
Le texte doit être restitué par les aides techniques. Mais il est ignoré à cause de l'utilisation d'une propriété incorrecte display:none pour afficher/cacher le texte.

Scénario d'un contenu masqué qui doit être ignoré :

Un menu, masqué avec la propriété opacity:0, s'affiche au clic sur un bouton le précédent.
Tant que le bouton n'est pas cliqué, menu doit être ignoré par les aides techniques, mais il ne l'est pas à cause de l'utilisation d'une propriété incorrecte opacity:0 pour afficher/cacher le menu.

Bouton pertinent : réaliser un intitulé de bouton

3 types de bouton de formulaire :

  • Balise <input> avec l'attribut type="submit", type="reset", type="button"
  • Balise <input> avec l'attribut type="image"
  • Balise <button> avec l'attribut type="submit", type="reset", type="button"

L'intitulé du bouton peut être :

  • Le contenu de l'attribut value de la balise <input> de type submit, reset, button.
  • Le contenu de l'alternative, l'attribut alt de la balise <input> de type image.
  • Le contenu de la balise <button>.
  • Le contenu du passage de texte associé au bouton via l'attribut aria-labelledby.
  • Le contenu de l'attribut aria-label.
  • Le contenu de l'attribut title.

Bouton pertinent : réaliser un intitulé de bouton

  • Passage de texte associé au lien avec l'attribut aria-labelledby.
  • Sinon, contenu de l'attribut aria-label.
  • Sinon, l'attribut alt de la balise <input> de type image.
  • Sinon, l'attribut value de la balise <input> de type submit, reset, button..
  • Sinon, contenu de la balise <button>.
  • Sinon, contenu de l'attribut title.

  • Un texte avant le champ "Les champs nom, prénom et courriel sont obligatoires", positionné au tout début formulaire.

Remarque concernant l'indication visuelle accompagnant l'attribut required ou aria-required :

Contrôle de saisie : aide de saisie

L'aide à la saisie sur un champ permet d'apporter à l'utilisateur une information sur la façon de renseigner le champ. Elle est nécessaire uniquement quand le champ demande une saisie dans un format spécifique.

Quelques exemples :

  • "Courriel (prenomnom@domain.com)"
  • "Code postal (00000)"
  • "Date (jj/mm/aaaa)" si le champ n'est pas de type date

Contrôle de saisie : aide de saisie

Lorsqu'un champ comporte un masque de saisie, l'aide de saisie ne doit pas être présente.

Le masque de saisie est géré par le navigateur et peut être différent selon les paramètres de l'utilisateur (langue, emplacement géographique).

C'est notamment le cas pour les types de champs suivants :

  • Type date
  • Type datetime
  • Type time

L'aide de saisie peut être implémentée de plusieurs façons :

  • Dans la balise <label>.
  • Dans un passage de texte visible associé au champ avec l'attribut aria-describedby.

À noter : L'utilisation de l'attribut aria-describedby doit être rempli avec l'id d'un passage de texte, l'instar de l'attribut aria-labelledby.
Il permet d'ajouter une description à l'étiquette du champ, à l'inverse de l'attribut aria-labelledby qui remplace l'étiquette du champ.

L'attribut placeholder ne permet pas de remplir la fonction d'aide de saisie.

Contrôle de saisie : erreur de saisie

L'absence de saisie ou la saisie incorrecte d'un champ doit être signalée à l'utilisateur par une information visible et doit permettre d'identifier nommément le champ.
Cela concerne tous les champs en erreur, même ceux qui ne sont pas obligatoire.

Le message d'erreur doit proposer un exemple de saisie réelle. Par exemple, pour un courriel, le message d'erreur pourrait être "veuillez renseigner un courriel valide, exemple : jean.dupond@gmail.com".

De la même façon que pour l'aide de saisie, le message d'erreur doit être indiqué par :

  • Dans la balise <label>.
  • Dans un passage de texte visible associé au champ avec l'attribut aria-describedby.
  • Avec l'attribut aria-invalid="true", accompagné d'un message d'erreur visible dans l'étiquette du champ.

Par défaut, les navigateurs fournissent une gestion très efficace des erreurs, mais ils ne comportent pas toujours d'exemple selon le navigateur. Utiliser l'API Constraint Validator pour les personnaliser, les rendre conformes et identiques pour tous les navigateurs.

Limite de temps

Critère 13.1. Pour chaque page web, l'utilisateur a-t-il le contrôle de chaque limite de temps modifiant le contenu (hors cas particuliers) ?

Les personnes ayant un handicap moteur ou cognitif ont besoin passer plus de temps sur une page pour réaliser une ou plusieurs actions.

Quelques exemples :

  • Un site d'information dont le contenu se rafraichit automatiquement et toutes les 5 minutes.
  • Un formulaire à remplir en moins de 5 minutes.

Au moins l'une des règles suivantes doit être respectées :

  • L'utilisateur peut arrêter ou relancer le rafraîchissement ;
  • L'utilisateur peut augmenter la limite de temps entre deux rafraîchissements de dix fois, au moins ;
  • L'utilisateur est averti de l'imminence du rafraîchissement et dispose de vingt secondes, au moins, pour augmenter la limite de temps avant le prochain rafraîchissement ;
  • La limite de temps entre deux rafraîchissements est de vingt heures, au moins.

À noter : lorsque l'accomplissement d'une tâche dans un temps imparti est essentiel, la règle n'est pas applicable. C'est le cas notamment lors d'un examen dans un temps imparti.

Critère 13.2. Dans chaque page web, l'ouverture d'une nouvelle fenêtre ne doit pas être déclenchée sans action de l'utilisateur. Cette règle est-elle respectée ?

L'ouverture automatique d'une nouvelle fenêtre est un changement de contexte qui peut perturber les utilisateurs s'ils n'en sont pas avertis. Les utilisateurs pourraient ne pas comprendre ce qu'il s'est passé et mettraient du temps à revenir sur la page qu'ils étaient en train de consulter.

À noter : il n'est pas question ici d'ouverture de page dans une nouvelle fenêtre ou un nouvel onglet, ni de fenêtres modales qui s'ouvrent automatiquement. Bien que ces pratiques soit conformes, elles sont aussi à éviter, à moins que le contenu soit d'une importance capitale.

Contenu en mouvement ou clignotant

Critère 13.8. Dans chaque page web, chaque contenu en mouvement ou clignotant est-il contrôlable par l'utilisateur ?

Lorsqu'un contenu est en mouvement, l'attention des personnes ayant des troubles de l'attention, comme de l'hyper activité, porte sur ce mouvement, les empêchant de consulter le reste de la page.

Ces éléments en mouvement doivent respecter les règles suivantes :

  • La durée du mouvement est inférieure ou égale à 5 secondes, sans répétition.
  • L'utilisateur peut arrêter et relancer le mouvement.
  • L'utilisateur peut afficher et masquer le contenu en mouvement.
  • L'utilisateur peut afficher la totalité de l'information sans le mouvement.

Gestes complexes

Critère 13.10. Dans chaque page web, les fonctionnalités utilisables ou disponibles au moyen d'un geste complexe peuvent-elles être également disponibles au moyen d'un geste simple (hors cas particuliers) ?

Les personnes ayant un handicap moteur peuvent rencontrer des difficultés à réaliser des gestes trop complexes comme des balayages. Dans le cas de gestes complexes, il est nécessaire de proposer aux utilisateurs des alternatives avec des gestes simples.

  • Geste complexe : geste impliquant plus d'un point de contact sur l'écran (zoom, dézoom), un geste basé sur le suivi d'une trajectoire ou un chemin (Drag and Drop).
  • Geste simple : geste impliquant uniquement un point de contact à la même place sur l'écran (simple clic ou pression, double clic ou double pression, ou clic ou pression prolongé).

À noter : il existe des cas particuliers notamment lorsque l'usage du geste complexe est essentiel comme la signature à main levé.

Annuler une action

Critère 13.11. Dans chaque page web, les actions déclenchées au moyen d'un dispositif de pointage sur un point unique de l'écran peuvent-elles faire l'objet d'une annulation (hors cas particuliers) ?

Certains utilisateurs qui ne peuvent avoir une précision suffisante (handicap moteur) peuvent déclencher une action sur un élément interactif par inadvertance. Afin d'éviter des actions accidentelles, il est nécessaire de permettre aux utilisateurs d'annuler une action.

Si l'activation ne se produit que lorsque le pointeur est relâché, les utilisateurs ont la possibilité d'abandonner l'activation en éloignant leur pointeur ou leur doigt de la cible avant de relâcher.

  • Réaliser les actions uniquement sur des événements relâché ou relevé (mouseup, touchend, click).
  • L'événement clic se déclenche lorsque le bouton principal de la souris est relâché et constitue un exemple d'événement relâché implicite.
  • Un glisser/déposer doit pouvoir être annulé en "déposant" dans une zone non réactive, ou via un bouton d'annulation.
